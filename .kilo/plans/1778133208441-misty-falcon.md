# Self-Hosted Firecrawl Product Fork Plan

## Direction
You are no longer trying to approximate Firecrawl Cloud from the outside.

You are building an independent, self-hosted product fork that:
- removes dependency on hosted Firecrawl agent infrastructure
- replaces Firecrawl-owned engine dependencies with your own acquisition stack
- exposes your stack cleanly to agents over MCP
- can be operated for a small number of paying clients
- keeps docs, release workflows, and auth/billing aligned with that reality

## Executive recommendation

### Product stance
Treat this as a hard product fork with three planes:
1. **Control plane**: tenant tokens, quotas, admin UI, MCP access policy
2. **Data plane**: the Firecrawl API/queue/scrape/extract/agent code in this repo
3. **Acquisition plane**: your own remote engine service for search, anti-bot handling, browser execution, and retries

### Core technical recommendation
Do **not** rip out every Fire-engine integration first.

Instead, build **your own internal engine service that speaks the current Fire-engine HTTP contract** so the existing API code can keep working while you replace upstream dependencies safely.

That gives you:
- a fast path to independence
- less churn in `apps/api`
- a clean seam for later renaming and deeper refactors
- the ability to improve the acquisition stack independently from API semantics

### Commercialization recommendation
Do **not** expose the current `USE_DB_AUTHENTICATION=false` bypass mode to real clients.

For paid client access, use your own principal/token/quota control plane. The strongest near-term candidate already exists in the sibling `fire-enrich` project.

## What the repo already tells us

### Confirmed hosted dependency gaps
- `apps/api/src/controllers/v2/agent.ts` is only a hosted passthrough.
- It requires `EXTRACT_V3_BETA_URL` and proxies to `/internal/extracts` with `AGENT_INTEROP_SECRET`.
- `apps/api/src/controllers/v2/agent-status.ts` and `agent-cancel.ts` are also hosted-oriented.
- Self-hosted `/v2/agent` parity is therefore blocked by hosted infrastructure, not routing.

### Existing local capabilities worth keeping
- `docker-compose.yaml` already runs a self-hosted MCP service pointed at `http://api:3002`.
- The API already has a full local `v1 /deep-research` pipeline:
  - `apps/api/src/controllers/v1/deep-research.ts`
  - `apps/api/src/lib/deep-research/*`
  - `apps/api/src/services/queue-worker.ts`
- Search already has local fallback paths:
  - `FIRE_ENGINE_BETA_URL`
  - `SEARXNG_ENDPOINT`
  - DuckDuckGo fallback
- Browser/session infrastructure already exists:
  - `v2 /browser*`
  - `v2 /scrape/:jobId/interact`
  - `apps/api/src/lib/scrape-interact/*`

### Existing control-plane seed
From `apps/nuq-postgres/fire-enrich.sql` and the sibling `fire-enrich` repo:
- there is already a `principals` table
- there is already a `usage_events` ledger
- there is already an admin surface for issuing bearer tokens with quotas and optional BYOK
- there are already MCP snippets in the ops UI for clients such as Claude, Cursor, and OpenCode

This is important because it is a more realistic base for “a few paying clients” than Firecrawl’s Supabase-centric team/org auth.

### Important traps
- The current auth path returns a mock unlimited account when `USE_DB_AUTHENTICATION !== true`.
- That is acceptable for internal/self-use but not for a public paid service.
- The adjacent MCP README documents `firecrawl_agent`, but current source/dist do not implement it.
- `README.md` and `SELF_HOST.md` still position the project as Firecrawl open source plus hosted cloud references, which will be misleading once you commercialize your fork.

## Build your own engine: recommended strategy

### Why emulate the current contract first
The current API already knows how to speak to a remote engine through:
- `POST /scrape`
- `GET /scrape/:jobId`
- `DELETE /scrape/:jobId`
- `POST /search`
- `POST /map`
- `POST /v2/search`

Those contracts are visible in:
- `apps/api/src/scraper/scrapeURL/engines/fire-engine/*.ts`
- `apps/api/src/search/fireEngine.ts`
- `apps/api/src/search/v2/fireEngine-v2.ts`

If your service matches those shapes, you can swap in your own engine with far less API churn.

### Engine roadmap

#### Phase E1: compatibility service
Implement a service that accepts the existing engine requests and produces compatible responses.

Minimum endpoints:
- `POST /scrape`
- `GET /scrape/:jobId`
- `DELETE /scrape/:jobId`
- `POST /search`
- `POST /map`
- `POST /v2/search`

Minimum backend capabilities:
- direct fetch mode
- Playwright mode
- async job queue for heavy scrapes
- screenshot support
- action execution support where feasible
- response normalization to current schemas

#### Phase E2: acquisition quality
Add the hard parts incrementally:
- proxy pool abstraction
- stealth browser mode
- TLS fingerprinting / alternative HTTP client
- retry classification by failure type
- domain-specific routing / forced engines
- per-domain success metrics

#### Phase E3: operator intelligence
Add features Firecrawl Cloud owns today but you can own yourself:
- domain-level engine policy store
- bad-domain heuristics and fallback promotion
- per-client usage shaping
- per-engine success dashboards
- anti-bot incident analytics

## Immediate answer: how your agents can access your Firecrawl now

### Current MCP path
Your agents can already access your self-hosted Firecrawl over MCP today.

There are two practical access modes:
- remote HTTP MCP via your deployed `/mcp`
- local stdio MCP pointed at your self-hosted API URL

### Best current auth model for clients
Use a principal token from the `fire-enrich` control plane, not a Firecrawl cloud key.

The sibling `fire-enrich` ops UI already shows the intended HTTP MCP shape:
- `Authorization: Bearer <principal-token>`
- Cloudflare Access service token headers for protected remote HTTP MCP

### Recommended MCP config shape
Remote HTTP example pattern:

```json
{
  "mcpServers": {
    "firecrawl-self-hosted": {
      "type": "http",
      "url": "https://crawl.theochinomona.tech/mcp",
      "headers": {
        "Authorization": "Bearer <principal-token>",
        "CF-Access-Client-Id": "<service-token-id>",
        "CF-Access-Client-Secret": "<service-token-secret>"
      }
    }
  }
}
```

Short-term usable MCP tools:
- `firecrawl_scrape`
- `firecrawl_search`
- `firecrawl_extract`
- `firecrawl_crawl`
- `firecrawl_research`
- browser tools if enabled

Do **not** promise `firecrawl_agent` yet until it is actually implemented in the MCP server.

## Implementation plan

### Track A: fork cleanup and product boundaries
Decide explicitly that this fork is self-hosted-first and not a thin wrapper around Firecrawl Cloud.

Required outcomes:
- no required dependency on `EXTRACT_V3_BETA_URL`
- no customer-facing dependency on Firecrawl cloud auth
- no public docs that imply hosted Firecrawl is required for core features

Recommended boundaries:
- this repo = data plane
- `fire-enrich` = control plane until/unless you absorb it
- new engine service = acquisition plane

### Track B: authentication and billing for paying clients
The current bypass auth path is not enough.

Recommended approach:
- use `fire-enrich` principals as the first customer identity model
- issue one bearer token per client
- meter usage with `usage_events`
- apply monthly pooled quotas
- allow BYOK as an upsell / escape hatch

Recommended near-term policy:
- do **not** expose `USE_DB_AUTHENTICATION=false` publicly
- front public traffic through the principal-bearing layer
- let the internal Firecrawl API remain private behind MCP/admin/gateway if needed

Optional future direction:
- later unify principal auth and Firecrawl team/org auth if you truly need native multi-tenant API keys inside `apps/api`
- do not make that a blocker for the first paid clients

### Track C: local `/v2/agent`
Build the agent feature locally and stop relying on hosted passthrough.

Recommended first engine:
- use the existing local deep-research pipeline as the first backend for `v2 /agent`

Why:
- already queued
- already iterative
- already search/scrape/analyze driven
- easier to ship than a brand new planner from zero

Implementation direction:
- add a self-hosted local agent state store in Redis
- make `v2 /agent` route into local execution when self-hosted mode is enabled
- keep `urls`, `schema`, `webhook`, and `model` semantics compatible where practical
- add cancellation and status tracking

### Track D: stop using hosted-style persistence for agent
Current `v2` agent status assumes hosted-style `agents` persistence.

Recommended replacement:
- add `apps/api/src/lib/agent/*`
- keep local agent job state in Redis first
- record:
  - principal/team ownership
  - status
  - progress
  - result payload
  - model tier
  - error
  - cancellation flag
  - expiry

Keep hosted fallback out of the default path.

### Track E: your own engine service
Create `apps/engine-service` or equivalent.

Recommended order:
1. Implement compatibility endpoints that match current Fire-engine usage.
2. Point `FIRE_ENGINE_BETA_URL` at your own service first.
3. Once stable, rename docs/envs to your own terminology.
4. Only later rename code paths if the compatibility seam becomes limiting.

This avoids a large simultaneous refactor.

### Track F: search independence
Hosted-quality research depends heavily on search quality.

Recommended search stack order:
1. Stand up reliable `SEARXNG_ENDPOINT` as the base search provider.
2. Add optional alternate providers behind your own engine service.
3. Add ranking, dedupe, freshness, and provider fallback in the engine layer.
4. Keep the API layer unaware of provider specifics.

### Track G: browser and anti-bot independence
Your custom engine should eventually own the hard acquisition layer.

Recommended acquisition stack:
- fetch for cheap/simple pages
- Playwright/browser cluster for JS-heavy pages
- proxy pool abstraction
- stealth / TLS / mobile proxy modes
- engine-forcing policy per domain

Use the existing concepts already visible in the repo:
- forced engine domains
- `stealthProxy`
- `mobileProxy`
- browser session infrastructure

### Track H: MCP truthfulness and usefulness
The MCP layer must reflect what is actually implemented.

Near-term MCP work:
- keep `firecrawl_research` as the self-hosted research tool
- add an explicit `firecrawl_deep_research` wrapper if useful
- document principal-token auth and Cloudflare headers

After local agent works:
- implement real `firecrawl_agent`
- implement real `firecrawl_agent_status`
- point them at your local API
- do not rely on README-only claims

### Track J: git workflow and release workflow
You asked for a real git workflow plan.

Recommended branch model:
- `main`: your product trunk
- `feat/*`: feature branches
- `fix/*`: bugfix branches
- `infra/*`: deployment / image / engine / ops work
- `docs/*`: docs-only changes
- `upstream-sync/*`: periodic upstream merge branches

Recommended rules:
- never commit directly to `developer`
- never ship from ad hoc local branches
- every substantial change lands through PRs into `main`
- upstream syncs happen through dedicated `upstream-sync/*` PRs, never mixed with product work

Recommended upstream workflow:
1. add upstream remote outside this plan if not already present
2. regularly branch from your `main` into `upstream-sync/YYYY-MM-DD`
3. merge or rebase upstream there
4. review conflicts for product assumptions:
   - hosted docs
   - cloud-only endpoints
   - billing/auth changes
   - engine integrations
5. merge into `main` only after verification

Recommended release workflow:
- tag release branches or semver tags from `main`
- publish separate images for:
  - API
  - MCP
  - engine service
  - optional control-plane web
- keep staging deploys manual until agent/engine stabilize

Recommended image namespace changes:
- stop publishing under upstream naming for your product builds
- use your own registry org/repo naming
- keep multi-arch only where it matters operationally

### Track K: CI/CD changes
Current workflows are upstream-oriented.

Recommended future workflow changes:
- split API, MCP, and engine-service CI
- add integration tests for principal-token auth path
- add agent snips for self-hosted local execution
- add engine-service contract tests against current API expectations
- add deployment workflows for your own registry/image names

### Track L: legal and commercial reality
This repo is AGPL-licensed.

That means commercialization is possible, but you should plan on:
- keeping corresponding source available for the network-served modified server components
- documenting what parts are your fork vs upstream
- avoiding any ambiguous “cloud parity” claim you cannot actually support

This is not just legal hygiene. It also helps your clients trust what is actually running.

## Final recommendation
The best path is not “strip all Firecrawl references now and rewrite everything.”

The best path is:
- keep the current seams that are useful
- replace the hosted dependencies behind those seams with your own services
- move client auth and quotas onto the existing principal model
- turn deep research into the first real self-hosted agent backend
- then clean up names, docs, and release workflows once the system is actually yours in operation

That gives you the fastest route to a real, sellable self-hosted product without forcing a high-risk all-at-once rewrite.
